企業實踐：誰在應用這套思維？

Spotify

Spotify 是「文件即代碼」與「架構視覺化」的先驅。他們透過自動掃描 (CaD) 數千個微服務的依賴關係，並將其定義為可視化的系統圖譜，解決了工程師在龐大舊系統中迷路的問題。

資料來源 Spotify Engineering Blog

Netflix

Netflix 發展出了一套即時拓撲視覺化思維。他們透過監控流量自動生成調用鏈路圖 (CaD)，讓架構師能一眼看穿哪些舊服務已經過載或應該被廢棄。

資料來源 Netflix Vizceral GitHub Netflix Tech Blog

Uber

在將單體應用拆解為微服務的過程中，Uber 大量使用自動依賴分析。他們將程式碼轉化為圖表以識別強耦合模塊，並將重構後的設計定義為文字藍圖 (DaC)，確保設計準則被嚴格執行。

資料來源 Uber Engineering Blog

GitHub

GitHub 將「Diagram as Code」原生集成到了 Markdown 預覽中。這推動了業界將架構圖直接寫在代碼庫旁的風潮，讓工程師能在提交代碼時，一併審查架構的變更。

資料來源 GitHub Blog: Mermaid in Markdown

核心概念視覺化對比

Code as Diagram

自動化現狀分析

透過分析技術直接從源碼產出。它能揭露人眼難以發現的「隱形依賴」。在考古階段，這能幫助你標記出那些需要被拆解的複雜模塊。

Diagram as Code

結構化架構定義

將理想架構撰寫為 DSL。這份「圖表代碼」就是你的施工藍圖，它與程式碼一同儲存在版本控制系統中。

1

CaD 逆向考古

自動掃描舊系統，產出原始依賴圖。暴露混亂邏輯的重災區。

2

精煉與轉化

將自動產出的雜亂圖表轉化為簡潔且具語義的文字描述。

3

DaC 重構設計

直接在文字描述中「畫圖」，重新定義模組邊界。

4

藍圖導引編碼

依據藍圖精確實現新架構。
